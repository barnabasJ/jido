# Task 0002 — Flatten `agent.state`; `path:` required; runtime identity on server struct

- Commit #: 2 of 9
- Implements: [ADR 0014](../adr/0014-slice-middleware-plugin.md) — path-based registration and runtime identity
- Depends on: 0001
- Blocks: 0003, 0004, 0005, 0006
- Leaves tree: **red** (many callsites broken until later commits)

## Goal

Remove every magic `:__*__` atom key from `agent.state`. Replace the `state_key:` agent option with a required `path:`. Move runtime identity (`partition`, `parent`, `orphaned_from`) off `agent.state` and onto `%Jido.AgentServer.State{}` as typed struct fields only. Add identity-transition signals.

**No on-disk migration**: per ADR 0014's "no external users exist," pre-refactor checkpoints don't need a forward-compat path. Any existing local-dev or test-fixture checkpoints are regenerated by running the post-refactor code from fresh. Keeps the code surface small; trades a one-time regeneration for no maintained migration shim.

This is the commit where the shape of `agent.state` changes from `%{__domain__: %{...}, __thread__: %{...}, __partition__: ..., ...}` to `%{<declared_path>: %{...}, thread: %{...}, ...}`.

### `lib/jido/agent/default_plugins.ex` — **not touched in this commit**

The default_plugins override map stays keyed by `state_key` (e.g., `%{__thread__: false}`) through C2, C3, and C4. It flips to path-keyed (`%{thread: false}`) in C5, paired with the plugin macro rewrite and in-tree plugin migrations.

Rationale: in-tree plugins still use the old `Jido.Plugin` macro (unchanged until C5), so they export `state_key/0` but not `path/0`. Deferring this flip keeps the default_plugins mechanism consistent with whatever the plugins actually export at any given commit.

## Risks

- `Directive.emit_to_parent/3` is deleted, not migrated. Previous users build `%Directive.Emit{dispatch: {:pid, target: ctx.parent.pid}}` themselves (with a nil-guard if they care about the orphaned case). C8 migration guide includes a 5-line reference snippet. Removes the implicit "silent `nil` on orphaned" behavior that was hard to reason about.
- `scheduler.cron_specs_state_key()` at [lib/jido/persist.ex:353, 544](../../lib/jido/persist.ex) is a separate reserved key for cron spec storage. With no migration pass shipping, this key continues to round-trip unchanged — verify hibernate/thaw preserves it against a fresh fixture.
- **`State.from_options/3` signature will extend in C4**: task 0004 threads `middleware_chain:` through this function as a 4th arg (keyword opts). C2 doesn't need to introduce the 4th arg yet, but don't freeze the function signature as "always 3-arg" in a way that forces a conflict in C4. Leaving the function as `from_options/3` with a TODO comment, or pre-emptively adding an `opts \\ []` tail, both work.
